# Midnight Network documentation guide

Midnight Network’s documentation provides developers with an entry point into the privacy-focused ecosystem. The documentation includes Getting started guides, API reference, tutorials, Compact language reference, and SDK documentation. 

This guide explains the documentation structure, along with the processes and workflows for creating, maintaining, and managing Midnight docs.

## Documentation architecture

The Midnight documentation site uses [Docusaurus](https://docusaurus.io/docs) and multiple independent `plugin-content-docs` instances rather than a single preset. This gives each major section its own URL namespace, sidebar configuration, and versioning behavior without cross-contamination.

### Section overview

These are the main sections where content is actively maintained:

| Section  | Source directory | URL prefix | Sidebar config |
| :---- | :---- | :---- | :---- |
| Main docs | docs/ | / (site root) | sidebars.main.js (autogenerated) |
| API reference | api-reference/ | /api-reference | sidebars.apis.js (autogenerated) |
| SDK docs | sdks/ | /sdks | sidebars.sdks.js (autogenerated) |
| Developer blog | blog/ | /blog | Uses default blog config, no sidebar file |

All three doc plugins use fully autogenerated sidebars. Category labels, positions, and link behavior come entirely from `_category_.yaml` files in each directory. This is fully explained in the [category creation](?tab=t.0#bookmark=id.s0ktrg85k9t8) section.

### Main documentation (docs/)

The primary docs directory covers the product overview, getting started, examples, multi-level tutorials, concepts, guides, Compact language reference, node operations, release notes, and troubleshooting. The section is versioned, with the current (labeled v1) as the live version and 0.0.0 as the archived legacy v0 content, served via main\_versioned\_docs/.

These are some of the design decisions and customizations introduced:

* Compact language docs are part of the main docs, not a separate plugin. An earlier configuration had Compact as its own plugin with a dedicated URL prefix (/compact). This was moved to docs/compact/, so Compact pages now live at the site root alongside the rest of the docs.  
* The release notes sidebar is intentionally flattened. The relnotes/ directory is organized into one subfolder per component (Node, Indexer, Wallet SDK, Compact, etc.), but the `sidebarItemsGenerator` in `docusaurus.config.js` strips nested subcategories, exposing only a flat list of pages under "Release notes". This gives readers a single chronological view rather than a component-per-component tree.  
* Underscore-prefixed folders (\_architecture, \_contribute, \_legal, etc.) hold auxiliary content not shown in the primary navigation. 

**Tip**: Prefix a file or directory with an underscore to hide it from the live site while preserving its content in the repository.

### API reference (api-reference/)

Auto-generated reference documentation for all JavaScript/TypeScript packages and the GraphQL Indexer API. 

The content is pulled through a GitHub Actions workflow from the component repository. The Midnight Indexer docs use the @graphql-markdown/docusaurus plugin to generate the schema reference docs. The Indexer schema is sourced from `static/midnight-indexer/schema-v4.graphql` and renders under `api-reference/midnight-indexer/`.

The section is intentionally read-only from an editorial standpoint. The docs are pushed from the source repo and schema files rather than being hand-authored.

### SDK documentation (sdks/)

The SDK docs include sections on core Midnight SDKs and a community section spotlighting ecosystem devtools.

### Developer blog (blog/)

The developer blog serves as the location for content not generally suited as documentation. For example, network announcements, hackathons, or community events.

## Documentation workflow automation

There are several automations and workflows built into the Midnight documentation repo to assist with routine tasks like reviewing contributions and checking for broken links. Below is a list of the tools and their use cases:

### Linting and style

[**Vale config**](https://github.com/midnightntwrk/midnight-docs/blob/main/.vale.ini): `.vale.ini`  
[**Midnight rules**](https://github.com/midnightntwrk/midnight-docs/tree/main/.github/styles/Midnight): `.github/styles/Midnight`  
[**Workflow source**](https://github.com/midnightntwrk/midnight-docs/blob/main/.github/workflows/vale.yml): `.github/workflows/vale.yml`

Vale integration adds an extra layer of review to the docs. It is preconfigured with rules from the Midnight brand style guide. The Vale integration aims to reduce review time by allowing contributors to lint and self-edit their work before submitting it for review.

Vale runs via a GitHub Actions workflow that checks only the updated docs and comment suggestions when a PR is opened.

These suggestions and flags help contributors self-edit their work before a reviewer takes a look. 

* **Vale config**: The .vale.ini file contains the Vale integration configuration. Without it, Vale won’t run on the repository. It tells Vale where to look for the brand style to use whenever it is called.  
* **Midnight brand styles**: The rules are located in the `.github/styles/Midnight` folder. It includes several rules that implement the [Midnight brand style guide](https://docs.google.com/document/d/1Srfgnn5Utp6fDXxJknwpQUd6l5TQNR9k_BJjtjAmm_s/edit?usp=sharing).  
* **Vocab**: The [`.github/styles/config/vocabularies/Midnight`](https://github.com/midnightntwrk/midnight-docs/tree/main/.github/styles/config/vocabularies/Midnight) contains two files: *accept.txt* and *reject.txt*. The *accept.txt* file contains dev-related and Midnight-specific terms that Vale should accept during reviews. On the other hand, *reject.txt* contains banned terms that must not appear in the doc content.  
* **Vale workflow**: The workflow file runs whenever a PR is opened on the docs repository. It runs the Vale check on the updated files.

The rules are evolving and are constantly updated, particularly when an issue is discovered with any of them. As you review PRs, update the vocab list with blockchain and Midnight-specific terms flagged as false positives by Vale.

The docs [README](https://github.com/midnightntwrk/midnight-docs/blob/f8980858cd989884db8aa9e27c0eaa2d775d9916/README.md?plain=1#L58) file includes instructions for installing and setting up Vale locally. It also includes details on how to run Vale checks locally to address issues before opening a PR.

### Broken link checker \- Lychee

[**Workflow source**](https://github.com/midnightntwrk/midnight-docs/blob/main/.github/workflows/broken-link-checker.yml): `.github/workflows/broken-link-checker.yml`

The broken link checker uses [Lychee](https://lychee.cli.rs/) to check for broken links, both internal and external. It checks only the updated or newly created doc to ensure links are working. The workflow checks both internal and external links.

There are instances where it returns false positives for some genuine links. But generally helps to keep an eye on the links included in a specific doc. 

No setup is required for this on the user's end. It runs via a GitHub Actions workflow and triggers whenever a pull request is opened against the main branch.

Once triggered, it checks the docs and comments on any broken links it catches.

### API docs workflow

[**Workflow source**](https://github.com/midnightntwrk/midnight-docs/blob/main/.github/workflows/apis.yml): `.github/workflows/apis.yml`

This GitHub Actions workflow automates copying and updating API reference docs from relevant component repositories into the API reference section of the docs.

The workflow includes a dispatch feature with a manual trigger from the GitHub repo. It should be run whenever any component in the API reference section is updated.

Follow the steps below to run the API docs workflow:

1. Switch to the Actions tab from the docs repo.  
2. Select the **Copy API Docs** workflow.  
3. Click the **Run workflow** button.   
4. In the overlay that appears, click the **Component docs to update** dropdown and choose the one you want to update.  
5. Specify the branch you wish to pull the changes from. For some components, the changes ship with the *main* branch. However, for components like the Midnight Ledger, you’ll need to specify the corresponding tag name as the branch, for example, *ledger-8.0.1*.  
6. After specifying the component to update and the branch, run the workflow. 

This creates a new PR with the updated API docs changes.

The files are usually much larger (100+). When reviewing, check that the site builds and deploys correctly. You can skip the style guide check for this since it’ll require updating multiple files.

In some rare cases, there might be a syntax error with the API docs. When this happens, inspect the build error and note the affected files. Then, create a fix branch and update the syntax error. Verify that everything is working as expected by running `yarn build`.

If the workflow fails to open a PR, then it is often due to an incorrect GitHub branch or repo URL specified when running the workflow.

### Midnight Indexer schema documentation

The Midnight Indexer provides a GraphQL schema for querying and retrieving blockchain data. Schema docs are automatically generated from the Midnight Indexer `schema.graphql` file. When you run the API docs workflow against the Midnight Indexer branch, it also downloads the updated schema file. After the PR is merged, the updated GraphQL schema is available in the [`static/midnight-indexer/`](https://github.com/midnightntwrk/midnight-docs/tree/main/static/midnight-indexer) directory.

To update the Midnight Indexer docs; 

1. Create a new feature branch  
2. Replace the schema address in the Indexer Schema Docs section in the [Docusaurus config file](https://github.com/midnightntwrk/midnight-docs/blob/82c762ca6167fa2250177861eeb2781b8d9a9497/docusaurus.config.js#L144) with the updated file path.  
3. Run the command below to generate the docs from the schema file:

| yarn docusaurus graphql-to\-doc |
| :---- |

4. Preview your changes locally to ensure everything renders correctly.   
5. After that, open a PR with the updated doc changes and merge to main.

**Note**: For more information, refer to the [official GraphQL-to-docs](https://graphql-markdown.dev/docs/get-started) guide.

### Deployment workflow

[**Build workflow source**](https://github.com/midnightntwrk/midnight-docs/blob/main/.github/workflows/build.yml): `.github.workflows/build.yml`  
[**Preview deploy workflow source**](https://github.com/midnightntwrk/midnight-docs/blob/main/.github/workflows/pr.yml): `.github.workflows/pr.yml`  
[**Production deploy workflow source**](https://github.com/midnightntwrk/midnight-docs/blob/main/.github/workflows/prod.yml): `.github.workflows/prod.yml`

These workflows manage the test and deployment pipeline of the docs site to Vercel. The build workflow ensures the site builds without errors. If it fails, then run `yarn build` locally to troubleshoot the error or view the workflow error log.

* **Build workflow**: The `build.yml` workflow runs the `yarn build` command. Its main purpose is to ensure that nothing breaks during site building.  
* **Preview deployment**: The `pr.yml` workflow deploys the build output to Vercel and comments a preview link to the PR. This way, contributors and/or reviewers can preview the site changes before merging them to the main branch. It also runs a regression test on doc URLs to ensure links aren’t broken.   
* **Production deployment**: The `prod.yml` workflow runs when the PR is merged into main, triggering a Vercel deployment. The credentials and environment variables needed for the preview and production deploy are maintained by the Midnight Foundation team (Chris Ferry and/or Lauren).

If the deployment workflow fails, then view the workflow's error logs. In some cases, it might be a build-related issue. If so, then run yarn build locally and troubleshoot any issues that prevent the site from building.

One of the leading causes of production deployment failures is incorrect credentials. Vercel credentials are added to the repo via GitHub secrets. If any of the API tokens are invalid or missing, then the deployment will fail. Make sure that the environment variables referenced in the workflow file are present in the GitHub repository secrets.

## Doc updates and maintenance

Because Midnight is an evolving blockchain, things change fast. The docs team needs to stay on top of it by ensuring the release notes and compatibility matrix are up to date.

The docs team gets pinged when a new component is released. In most cases, you might also receive details of the required changes for the compatibility matrix.

After updating the compatibility matrix, review the tutorials and troubleshooting docs to verify the component versions they’re running on. If it’s a minor release, update the package versions in the doc. 

However, if there are breaking changes in major releases, then you must update the affected docs and test to ensure the tutorial runs correctly.

After publishing the release notes, run the API docs workflow to update all API references.

### Release note process

Teams write release notes for the components they’re responsible for. For example, the Shielded team responsible for the [Midnight.js](http://Midnight.js) package produces the release notes for each component release. 

However, the docs team needs to take the notes and convert them into a doc-ready format, following the style guide and [release note template](https://github.com/midnightntwrk/midnight-docs/blob/main/RELEASE_NOTE_TEMPLATE.md).

Release notes for each component reside in the relnotes/\<component\> directory. Within each folder, create a new release note doc for each new release.

The main index page for the component (for example, relnotes/node.mdx) provides a brief overview and a dynamic list summarizing each release.

Using the Midnight node release as an example, you’ll find its dynamic list component in the [`src/components/DynamicListNode.js`](https://github.com/midnightntwrk/midnight-docs/blob/main/src/components/DynamicListNode.js) directory. You’ll also see the dynamic list for other components here.

When updating the dynamic list, the new release uses the “LATEST” tag. For older versions, use:

* “SUPPORTED” if the version is still compatible with other environments like Preview or Preprod  
* “UNSUPPORTED” if the version is still active but not recommended for any environment  
* “DEPRECATED” if the version or component has been fully sunset or works with a deprecated environment like Ledger v6 or v7 without backward compatibility.

## Draft a doc

Docs use the MDX file extension to support JSX components within a doc file. When creating a new doc, write the file name to match the doc's title and use the .mdx extension.

Define the doc frontmatter, which includes the title, license information, sidebar label, and position. 

The main doc should be written in accordance with the brand style guide. 

### Creating a new category

Categories are subfolders within the parent docs folder. Each new category requires a `_category_.yml` file to format how it renders.

You can use a generated index page or specify a custom one by adding the doc page's ID to the category config file.

### Generated index

This creates an index page that displays all the docs within the category in a grid.

| position: 20label: 'Error reference'collapsible: truecollapsed: truelink: type: generated-index title: Error reference description: Error reference for the Midnight API. |
| :---- |

### Custom index page

This is useful when you have an existing index page and need to use it for the category index:

| position: 10label: 'Overview'collapsible: true \# make the category collapsiblecollapsed: true \# keep the category closed by defaultlink: type: doc id: indexcustomProps: description: Overview of the Midnight APIs. |
| :---- |

The position in the category config defines where it resides in the sidebar navigation. It must be set to align with the information architecture.

The sidebar position of docs within a category is important and should align with the user’s journey. Use the `sidebar_position` frontmatter to control the position of docs. 

**Tip**: Use a double-digit range when specifying the range, like 10, 20, 30… This way, you can use a weight of 15 in the future to add a doc between 10 and 20\. If you use single digits, like 1, 2, 3, ..., then adding a new page in the middle of the list will require updating all the pages after it.

For more information on category metadata, see the [official Docusaurus](https://docusaurus.io/docs/sidebar/autogenerated) documentation.

## Unpublish or hide a page or folder

When you need to unpublish a blog, prepend an underscore to the file name. For example, if the doc file name is quickstart.mdx, rename it to \_quickstart.mdx. 

This excludes the file from the Docusaurus build. It makes sense to use the approach rather than deleting the file entirely, because then the page can be included in the next build when needed by simply reverting the filename change.

To unpublish an entire folder, use the same approach and rename the folder to include an underscore in its name.

## Versioning the docs

The goal of the versioning is to have a stable version of the docs while major changes are being worked on in the Canary branch.

During the doc site restructuring, doc versioning was implemented using the built-in [Docusaurus doc versioning CLI command](https://docusaurus.io/docs/versioning). 

The primary reason was to preserve historical data while implementing the revised information architecture. The timing was also perfect, given the migration from Testnet-02 to the Preview environment.

The genesis versioned doc is tagged v0.0.0. The goal is to align the docs with each major Midnight Ledger release. So that when there is a major release, the docs are versioned as well.

To version the doc, run the command below:

| yarn docusaurus docs:version DOC\_VERSION |
| :---- |

Be sure to replace DOC\_VERSION with the version number you intend to create, for example, 1.0.0. After running the command, it creates a new versioned docs in the main\_versions\_docs directory. This directory houses the current version of the doc.

After versioning the docs, add the new version to the Docusaurus config file under the main docs plugin configuration.

### Multiple docs plugin instances

For sites like the Midnight docs that have multiple instances of the docusaurus-docs-plugin configuration, the command above will likely not work. You must specify the ID of the plugin instance you want to version. Here is a list of available plugins Midnight Docs uses and their plugin ID:

| Plugin | Path | ID |
| :---- | :---- | :---- |
| Main documentation | docs/ | main |
| API reference | api-reference/ | apis |
| SDK documentation | sdks/ | sdks |

To version the main docs, use the command below:

| yarn docusaurus docs:version:main DOC\_VERSION |
| :---- |

In most cases, you’ll only need to version the main docs because that’s where content changes and decay rapidly. However, if you need to version other content types, then replace main with the plugin ID you want to version.

### Configure version behavior

The "current" version is the folder name in the `/docs` directory. For Midnight docs, typically, this would be the “Canary” branch, where changes for the new release are being worked on. This means the docs in the `/docs` folder will be accessible by adding /next/doc-title to the doc's URL. 

To implement this behavior, replace the value of the `lastVersion: "current"` line in the main docs plugin config with the most recent version number.

This means the versioned doc remains the main doc served to users. You can make changes to the main docs folder, which currently serves as the Canary version.

Once changes are complete and the release is ready, revert the changes made to the lastVersion line. This makes the docs folder the live version of the docs and removes the Canary version entirely.

Users can still access the versioned docs by adding the version number to the URL, for example, docs.midnight.network/0.0.0/getting-started. They’ll see a legacy flag for the versioned docs, indicating that the content is deprecated and no longer maintained.

**Note**: For more details, refer to the [Docusaurus versioning docs](https://docusaurus.io/docs/versioning).

## Known issues and gaps

The release notes process could be improved. There is a GitHub Actions workflow (`.github/workflows/release-notes.yml`) to copy the release notes from the relevant component repo to the Midnight docs. However, the workflow isn't fully usable because some release notes aren’t 100% ready for documentation. The notes required manual editing after merging the PR, which is inefficient.

Also, the processes for generating the Midnight Indexer schema docs rely on a manual trigger command to generate the docs locally and then open a PR with the updated docs. In the future, it should be streamlined to run the schema docs build command whenever the `schema.graphql` file is updated.

## AI agent docs friendliness checker

With the recent trend in AI, the primary consumers for docs are shifting from human developers to AI agents. This means the docs site should be optimized properly for AI agents. Fortunately, most of what makes documentation useful to agents also makes it more useful to humans.

Use the [AFDocs](https://afdocs.dev/) tool to run a check on the Midnight doc:

| npx afdocs check https://docs.midnight.network \--format scorecard |
| :---- |

It includes suggestions for issues it flags to help with improving the score.
